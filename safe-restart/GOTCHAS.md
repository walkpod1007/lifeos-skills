- [0.5] **現象**：macOS Sequoia (15.x) Mac 重開機後，`/usr/sbin/cron` 的 Full Disk Access 授權**會 stale invalidation 而失效**，但系統 UI 上看 cron 還在 FDA 列表裡（toggle 還勾著）。所有 user crontab job 訪問 `~/Documents/`、`~/Desktop/`、`~/Downloads/` 全部 silent fail，spawned bash 噴 `Operation not permitted`。**Apple 沒承認這是 documented 行為**，是 TCC daemon attribution 的 undocumented edge case。 **這次怎麼踩到的（2026-05-01）**：使用者問「為什麼 LINE 不工作」，發現 mac 04-29 21:40 重開機後 6 條 LINE supervisor 沒被自動拉起。我們昨晚（04-29）裝了 cron @reboot + `*/5` watchdog，**直到 04-29 23:11 都還能跑**——然後 cron 全部突然死掉，**33 小時無聲死亡**，沒任何告警。`/tmp/claude-line-watchdog.log` 塞滿 `Operation not permitted`，但沒人看 log 不會發現。 **修復方法**（治標）：System Settings → Privacy & Security → Full Disk Access → 找 `/usr/sbin/cron` → **toggle 關 → 開**（重新驗證）。60 秒內下個 cron tick 會自動恢復。**不需要 kickstart cron**（會被 SIP 擋且沒必要）。 **根治方向**（治本，已落地）： 1. **watchdog 搬出 `~/Documents/`** → `~/bin/life-os-watchdog/`，cron 失 FDA 也能讀到 .sh 2. **FDA canary 健康檢查**：watchdog 第一步 cat `~/Documents/life-os/.fda-canary`，失敗代表 FDA 死 3. **launchctl asuser osascript notification**：cron 直接跑 osascript **不會跳到 GUI**（cron 不在 user GUI bootstrap），要 `/bin/launchctl asuser "$(id -u)" /usr/bin/osascript -e ...` 4. **30 分鐘 rate limit + 復原時清 stamp**：避免 spam，又確保 FDA 復原後再失效時能再次告警 **為什麼這次重要**： - 「無聲失敗」是最壞模式——用戶不知道、log 沒人看、業務（LINE 訊息消費）默默積壓 - 33 小時的訊息積壓只在使用者主動問「LINE 不工作了」時才被發現 - 有告警系統 → 重開機後 60 秒內知道；沒有 → 看你哪天想到要 grep log **判斷觸發詞**： - 「cron 突然全死」「重開機後排程 job 沒跑」「launchd / cron Operation not permitted」 -
